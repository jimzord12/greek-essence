## 13. Prototype Flows and Scenarios

The clickable prototype must include the following end-to-end flows.

### Flow A — First-time destination-led inquiry

Home → Destinations → Paros & Antiparos → related journey → `Plan My Trip` with context → four steps → one validation correction → submission → confirmation.

### Flow B — Experience-led family inquiry on mobile

Mobile Home → menu → Experiences → Family Travel → related destination → `Plan My Trip` → conditional children/ages fields → completion or recoverable submission error → retry → confirmation.

### Flow C — Journey evaluation

Journeys → Journey Detail → route and customization explanation → related destination → return → personalize journey → form with journey context.

### Flow D — Trust and secondary contact

About → attributed hospitality/team evidence → How It Works → Contact → choose consultation or submit short general inquiry → success.

### Flow E — FAQ objection resolution

Landing/detail page → FAQ → budget/planning-process question → contextual Plan My Trip action.

### Flow F — Bilingual continuity

English destination detail → switch to Greek equivalent → navigate to Greek journey/detail → switch back while preserving equivalent context. Include a missing-translation alternative state.

### Flow G — Navigation and recovery

Compact menu open/close and nested navigation → not-found or unavailable content → recovery to relevant index → continue exploration.

### Flow H — Form resilience

Enter data → Back/Continue persistence → invalid date/email/group state → corrected values → simulated submission failure with preserved data → retry → success.

### Prototype content dataset

The prototype should include:

- all four priority destination cards and at least one complete destination detail;
- a representative `More of Greece` state;
- all confirmed experience themes as cards and at least one detail;
- at least three representative journey cards and one complete journey detail, clearly marked provisional until operational approval;
- one approved-style team profile and one attributed trust-item pattern using non-assertive placeholder labels if facts are pending;
- representative English and Greek navigation, key screens, form labels, errors, and confirmation;
- all significant component states required by the flows.
